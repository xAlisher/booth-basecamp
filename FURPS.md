# Booth — Functional Scope (FURPS+)

> **Status:** v0.2.1 shipped (broadcast-only · universal API · design-system UI)
> **Scope:** Broadcasting only — hosting a station. Listening is the [Receiver](https://github.com/xAlisher/receiver-basecamp)'s job.
> **Target:** Logos Basecamp (Linux x86_64), testnet-era platform; modules `radio_module` 0.1.0 + `radio_ui` 0.2.1
> **Non-goals (this version):** in-app listening · video display · a P2P media swarm · bundled runtime binaries

---

## Functionality (F)

### F1. Start / manage a station
- **F1.1**: Start a station with `visibility: public | private` and `privacy: onion | lan` (default `onion`).
- **F1.2**: Booth spawns a MediaMTX origin and mints an ingest URL (WHIP/RTMP/SRT) + stream key.
- **F1.3**: Stop a station; identity (path + key) persists across stop/start.
- **F1.4**: Station is persisted (`station.json`) and **auto-resumes on launch** — except Keycard identities (F3.3).

### F2. Discovery announce over Logos Messaging
- **F2.1**: Announce a JSON station record over `delivery_module` on the public directory topic `/radio-basecamp/1/directory/json`, or an unguessable private per-stream topic.
- **F2.2**: Re-announce every 15 s; announce is gated on the origin actually receiving a stream.
- **F2.3**: Announce carries the play URL — a `.onion` URL in onion mode (no IP), a LAN URL in LAN mode.

### F3. Station identity (secp256k1)
- **F3.1**: **Anonymous** — unsigned announce (`v:1`).
- **F3.2**: **Autogenerated** — device-local persisted secp256k1 key; announces signed (`v:2`), carrying pubkey + signature; human-anchored by a 3-word PGP fingerprint.
- **F3.3**: **Keycard** — hardware-derived key (`deriveKey("bc:radio")`), same identity on any device holding the card; does not auto-resume.

### F4. Ingest (host-operated, external)
- **F4.1**: Accept audio from **OBS Studio** (live) over WHIP/RTMP/SRT.
- **F4.2**: Accept audio from **Liquidsoap** (headless automated DJ).
- **F4.3**: Publishing requires the stream key; HLS read is public.

## Usability (U)
- **U1**: Single-panel `ui_qml` UI on the Logos design system; start/stop, identity tier, privacy mode, ingest details, live status.
- **U2**: First-launch **dependency preflight** — detects `mediamtx`/`tor`/`ffmpeg`/`torsocks`, shows a copy-able install command.
- **U3**: 3-word fingerprint shown for out-of-band station verification.
- **U4**: Headless operation supported (Liquidsoap + auto-resume under systemd) for always-on stations.

## Reliability (R)
- **R1**: Auto-resume re-spawns MediaMTX + tor and re-announces after a restart (F1.4).
- **R2**: Announce never falls back to a LAN IP in onion mode (no IP leak), gated on Tor descriptor publish.
- **R3**: Known caveat — a Tor HS descriptor can go dark on flaky HSDir uploads; a station restart republishes (#38/#46).

## Performance (P)
- **P1**: ~10 s HLS latency (acceptable for broadcast radio).
- **P2**: Audio-first, low bitrate — fits Tor's bandwidth.
- **P3**: Origin uplink is the scaling limit (direct per-host delivery; no swarm in v1).

## Supportability (S)
- **S1**: Two-module split (`radio_module` core + `radio_ui` universal UI) — core is headless-testable via `logoscore`.
- **S2**: Dual-licensed MIT OR Apache-2.0; releases signed by xAlisher.
- **S3**: Cross-platform listener companion (Receiver) ships on Linux, macOS, and Android.

## + (design constraints)
- **+1**: No central directory, no account, no server between host and listener.
- **+2**: `delivery_module` pinned to **v0.1.1**, installed separately (not platform-bundled); newer builds drift the IPC signatures.
- **+3**: Runtime binaries (MediaMTX/tor/ffmpeg) not bundled — blocked on `logos-module-builder#114`.
- **+4**: No blockchain — "announce" is a Logos Messaging heartbeat, not an on-chain record.
